# 素人种草运营工作台 · 实现规划 v1（交付 GPT 实现）

> **给实现者（GPT）的话**：这是一份系统实现规格，按本文实现/修改即可。系统是**单端网页**，唯一使用者是**内部运营工作人员**；兼职完全在系统外（靠人工对接），**不要做兼职端**。同目录的 `素人种草运营工作台.html` 是一个粗原型，可作 UI 手感参考，但**以本规格为准**。内容种子（吸脂剧本骨架、标准化素材清单、医美红线词库）在 `素人种草系统_方案文档_v0.1.md` 的第 5/6/11 节，建库时从那里取。

---

## 0. 范围与非目标

- **做**：内部运营用的网页工作台——案例归档、内容计划（含**随机生成**）、复制派发、执行核对、账号数据。
- **不做**：兼职端 / App；不做抖音自动发布（兼职手动发）；v1 不强制接 LLM（随机生成先用"内容池 + 规则组装"，LLM 作为可选增强）。

---

## 1. 定位与使用者

- **一句话**：内部运营工作人员用来**批量管理兼职 + 派发种草内容 + 核对执行 + 跟踪数据**的网页工作台。
- **唯一使用者**：内部工作人员（可能多人）。兼职看不到系统。
- **账号 : 案例 = 1 : 1**：一个案例 = 一个兼职 = 一个抖音号，有完整生命周期。

---

## 2. 核心工作闭环（系统要支撑的动作）

```
新建案例(归档兼职信息) → [随机生成整套内容计划] → 工作人员逐条复制 → 微信发给兼职
   → 兼职发完汇报 → 工作人员打开抖音主页核对真发了没 → 标记已核对 → 录入/同步账号数据
```

---

## 3. 页面结构（3 个视图）

**① 今日工作台（默认首页）**
- 跨所有案例，聚合**今天待派发**的内容（按案例分组，每条带[复制]按钮）。
- 聚合**待核对**的内容（兼职已汇报、等工作人员确认）。
- 目的：工作人员每天打开这一页就能干完日常，不用逐个案例翻。

**② 案例库**
- 所有案例的卡片/列表：兼职微信昵称、抖音号、项目、当前阶段、今日待派数、待核对数、最新粉丝数。
- 「+ 新建案例」入口。点卡片进详情。

**③ 案例详情**
- 头部：兼职微信昵称（可编辑）、抖音号、抖音主页链接（可点开核对）、项目、阶段、人设、起始日期。「编辑信息」「删除」。
- Tab A **内容计划**：表格列出每条（日期 / 用哪个号 / 类型 / 标题+正文 / 状态 / 操作）。
  - 顶部按钮：**[随机生成整套计划]**、**[+ 手动加一条]**。
  - 每条操作：[复制]、[兼职已汇报]、[打开抖音核对]→[已核对]、[换一条(重roll)]、[改]、[删]。
- Tab B **账号数据**：录入/展示粉丝、播放、点赞、评论，自动算较上次涨粉。（数据来源见第 8 节）

## 4. 数据模型

> 语言无关，下面用 TS 风格标注类型，GPT 按所选技术栈落地。

```ts
Case {
  id: string

  douyinId: string          // 抖音号
  douyinUrl: string         // 抖音主页链接（核对用）
  staff?: string            // 负责的内部工作人员
  project: string           // 项目类型，如「吸脂」（关联剧本模板）
  persona: Persona          // 人设（驱动随机生成的变量）
  stage: Stage              // 当前阶段
  startDate: string         // 起始日（YYYY-MM-DD，时间线锚点）
  plan: PlanItem[]
  metrics: Metric[]
  createdAt: string
}

Persona {                   // 随机生成时做变量替换 + 决定养号选题边界
  nickname?: string
  city?: string
  age?: number
  occupation?: string
  motivation?: string       // 做这个项目的动机故事（决策期内容钩子来源）
  tone?: string             // 说话风格（影响洗稿/语气）
}

  id: string
  date: string              // 发布日期
  time?: string             // 建议发布时刻（时间窗内随机）
  account: string           // 用哪个号（默认 case.douyinId，可填测试号）

  sourcePoolId?: string     // 来自哪条内容池模板（用于去重统计）
  note?: string
  reportedAt?: string
  verifiedAt?: string
}

Metric { id, date, fans?, plays?, likes?, comments?, note? }
```

## 5. ★ 随机生成引擎（本次重点）

### 5.1 它解决什么
- **一键成稿**：点新建案例 / 点[随机生成]，系统按项目+人设+起始日，**自动生成一整套带日期的内容计划**（种草帖 + 养号帖交错），工作人员只需微调+复制，不用手写。
- **去同质化（核心价值）**：几十上百个兼职都做吸脂，内容若雷同 → 平台判矩阵营销号 → 限流/封号。随机化 = 让每个案例的计划都不一样，是去同质化的主手段。

### 5.2 输入
`generatePlan(project, persona, startDate, options?)`
- `project`：决定用哪个剧本模板
- `persona`：变量替换 + 养号选题边界
- `startDate`：时间线锚点
- `options`：可覆盖默认节奏（养号:种草配比、每日发布时间窗、是否启用 LLM 洗稿等）

### 5.3 两个核心数据（详见第 6 节）
- **剧本模板 scriptTemplate**：内容计划的骨架（按项目）——有哪些阶段、各阶段时间窗/帖子数/配比/从哪些池抽。
- **内容池 contentPool**：一堆打了标签的可复用内容条目（标题模板 + 正文模板 + 变量占位符）。

### 5.4 生成算法（伪代码，可直接实现）
```
function generatePlan(project, persona, startDate, options):
    tpl   = scriptTemplates[project]
    rules = merge(tpl.pacing, options)        // 节奏参数
    plan  = []
    for stage in tpl.stages:
        n     = randInt(stage.countMin, stage.countMax)        // ① 帖子数随机
        dates = scatterDates(startDate, stage.dayFrom, stage.dayTo, n, rules.minGapDays)  // ② 日期在窗内随机分布、带最小间隔
        for d in dates:
            type    = weightedPick(stage.typeRatio)            // ③ 种草/养号按配比随机
            pool    = poolsMatching(project, stage.name, type)
            item    = pickFromPool(pool, {                     // ④ 按权重抽一条，去重
                          excludeRecentlyUsedAcrossCases: true,
                          decayByUsageCount: true })
            content = fillVars(item.contentTemplate, persona)  // ⑤ 变量替换 {城市}{年龄}{昵称}{项目}…
            content = rewrite(content, persona, rules.llm)     // ⑥ 洗稿（规则或LLM，见5.7）
            time    = randTimeIn(rules.publishWindows)         // ⑦ 当天发布时刻随机
            plan.push(PlanItem{ date:d, time, account:case.douyinId,
                                type, format:item.format, title:fillVars(item.title,persona),
                                content, status:'wait', sourcePoolId:item.id })
            bumpUsage(item.id, case.id)                        // 记录使用，供去重
    plan = sortByDate(plan)
    plan = complianceScan(plan)                                // ⑧ 红线词扫描（第9节）
    return plan
```

### 5.5 随机的维度（保证每个案例都不同）
| 维度 | 怎么随机 |
|---|---|
| 抽哪条文案 | 内容池按权重随机 + 跨案例去重（见 5.6） |
| 每阶段帖子数 | `countMin~countMax` 内随机 |
| 帖子日期分布 | 阶段时间窗内随机散布 + 最小间隔，避免同日扎堆 |
| 每天发布时刻 | `publishWindows`（如 10-12 / 19-21 点）内随机 |
| 种草/养号排布 | 按 `typeRatio` 概率决定每条类型，序列自然交错 |
| 文案措辞 | 洗稿层产出变体（见 5.7） |

### 5.6 去重 / 去同质化保证
- 内容池每条记 `usageCount` + `lastUsedCases[]`。
- 抽取时：① 硬排除「最近 N 个案例用过」的条目；② 用 `权重 = baseWeight / (1+usageCount)` 降低高频条目命中率。
- 变量替换（persona 不同）+ 洗稿层，进一步把同模板产出的文案拉开。
- 池子告急（可选条目不足）时，降级允许复用但强制走洗稿。

### 5.7 洗稿层 `rewrite()`（去同质化关键，分两档实现）
- **v1 规则档**（无依赖）：同义词替换、句子顺序微调、开头/结尾句式模板轮换、emoji/标点风格随机。
- **增强档（可选 LLM）**：把内容池条目当"种子提示词"，调用 LLM 按 persona 真实改写 + 控制语气。**实现时把 `rewrite()` 和 `pickFromPool()` 设计成可替换的抽象**，v1 走规则，后续无痛切 LLM。

### 5.8 重新生成 / 局部重 roll
- **整套重生成**：换随机种子重跑 `generatePlan`（已派发的条目可选保留）。
- **单条换一条**：只对该 slot 重新 `pickFromPool + rewrite`，日期不变。
- **锁定**：已 `report`/`ok` 的条目不被重生成覆盖。

## 6. 内容池 & 剧本模板（数据结构 + 种子来源）

**剧本模板（按项目，可后台编辑）**：
```json
{
  "project": "吸脂",
  "pacing": { "publishWindows": ["10:00-12:00","19:00-21:00"], "minGapDays": 1 },
  "stages": [
    { "name":"起号期",   "dayFrom":-14, "dayTo":-8, "countMin":4, "countMax":6, "typeRatio":{"养号":1.0} },
    { "name":"决策期",   "dayFrom":-7,  "dayTo":-1, "countMin":2, "countMax":3, "typeRatio":{"种草":0.7,"养号":0.3} },
    { "name":"术后恢复期","dayFrom":1,   "dayTo":30, "countMin":5, "countMax":8, "typeRatio":{"种草":0.5,"养号":0.5} },
    { "name":"成果期",   "dayFrom":31,  "dayTo":45, "countMin":1, "countMax":2, "typeRatio":{"种草":1.0} },
    { "name":"收尾期",   "dayFrom":46,  "dayTo":60, "countMin":1, "countMax":3, "typeRatio":{"种草":0.4,"养号":0.6} }
  ]
}
```

**内容池条目（一条一对象，可后台增删）**：
```json
{
  "id": "xz-decision-001",
  "project": "吸脂",
  "stage": "决策期",
  "type": "种草",
  "format": "口播",
  "titleTemplate": "纠结了大半年，{城市}的我到底要不要去抽脂",
  "contentTemplate": "{年龄}岁，{occupation}，体重不算重但{部位}就是下不去……说说我下决心前最纠结的3件事。",
  "vars": ["城市","年龄","occupation","部位"],
  "baseWeight": 1.0,
  "usageCount": 0,
  "lastUsedCases": []
}
```

**种子内容从哪来**：从 `素人种草系统_方案文档_v0.1.md` 取——
- 剧本骨架/阶段定义 → 第 6 节「种草剧本骨架」（说服任务 + 时间线 + 帖子数 + 钩子示例）。
- 吸脂各阶段必备素材/形式 → 第 5 节「标准化案例模板」。
- 养号池选题 → 第 8 节（理想接飞瓜/蝉妈妈爆款监测，v1 先手填一批通用选题）。
- 红线词库 → 第 11 节，喂给 `complianceScan()`。

> 内容池应**可在系统内编辑扩充**（增删条目、调权重），这样越用越丰富、去重越好使。

## 7. 状态机（PlanItem.status）

```
wait(待发) ──工作人员复制派发、兼职回报已发──▶ report(已汇报待核对)
report ──工作人员打开抖音核对真发了──▶ ok(已核对)
report/wait ──未发/异常──▶ skip
ok ──撤回──▶ wait
```
- 「今日工作台」据此聚合：`date==今天 && status==wait` 进待派发区；`status==report` 进待核对区。

## 8. 抖音监测 / 数据同步（务必按现实设计，预留接口）

抖音**没有免费、干净的公开 API** 抓任意账号内容/数据。三条路，让系统把"取数"抽象成一个 `DouyinDataSource` 接口，v1 实现"手动档"，其余作为可插拔实现：

| 路线 | 说明 | 取舍 |
|---|---|---|
| **① 半自动（v1 默认）** | 工作人员贴主页/作品链接，系统记录；核对靠点开链接人工确认；数据手动录入 | 零依赖、零成本、今天能用 |
| ② 抖音开放平台授权 | 兼职 OAuth 授权一次，拉自己账号的发布记录+数据 | 最准，但要兼职配合 + 开发对接 |
| ③ 第三方数据平台 API | 飞瓜 / 蝉妈妈 / 新抖 | 省自研，但有订阅成本、未必覆盖小号 |

- **接口抽象**：`DouyinDataSource { listPosts(account), getMetrics(account) }`，v1 返回手动录入数据，后续换②/③不改上层。
- 「核对真发了没」v1 = 打开主页人工看 + 标记；自动核对依赖①以外的路线。

---

## 9. 合规校验 `complianceScan()`

- 生成 / 保存内容时，扫描**医美广告红线词**（词库见 v0.1 第 11 节：绝对化用语、疗效/安全承诺、对比贬低、诱导限时等），命中则**高亮 + 拦截或提示替换**。
- 凡 `来源=生成` 的**结果/对比类**素材：默认禁止"伪造本人术后效果当真实发布"（用户已知此为高风险项；建议做成默认关闭的硬开关）。
- 这一步是发布前闸门，挂在 PlanItem 保存和"复制"动作前。

## 10. 非功能需求

- **存储**：v1 可纯前端 + 浏览器本地存储（localStorage/IndexedDB），并提供**导出/导入 JSON 备份**。
- **多人共用**：若多个工作人员要共享同一份数据，需轻量后端（任意栈 + 一个库即可），把上面的数据模型搬到服务端。GPT 按需选择；**数据模型不变**。
- **可维护**：剧本模板、内容池、红线词库都应**可在界面里编辑**，不写死在代码里。

## 11. 实现建议 & 分期

- **技术栈**：交给 GPT，约束是「轻」——单页应用即可，能先纯前端跑起来；要多人再加后端。别上重框架全家桶。
- **分期**：
  - **v1**：3 个页面 + 数据模型 + 随机生成（规则档）+ 手动数据 + 合规扫描 + 本地存储/备份。
  - **v1.1**：内容池/模板/红线词的后台编辑界面；单条重 roll、整套重生成。
  - **v2**：接 LLM 洗稿增强；接抖音数据源（②或③）；多人后端。
