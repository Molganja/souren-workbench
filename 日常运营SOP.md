# 素人系统日常运营 SOP

这份 SOP 面向每天实际使用系统的工作人员，也给 Codex/本地 clude 做交接参考。

## 1. 每天启动

1. 双击 `启动素人工作台.command`。
2. 打开 `http://127.0.0.1:5174/`。
3. 进入「系统配置」，先看「系统验收清单」：
   - `已就绪`：可以正常使用。
   - `待接入`：通常是 `IMAGE_API_KEY` 或本地 `clude/Claude`，不影响排期、文案、交付、核对。
   - `需处理`：优先处理，例如 GitHub main 未同步、目录异常。

如果要用桌面窗口版，在终端运行：

```bash
cd /Users/licc/Desktop/素人系统/app
npm run client
```

生成 Mac 客户端目录：

```bash
npm run client:mac
```

Mac 客户端会用 Electron 内置 Node 启动本地后端，不需要工作人员手动启动服务。普通运营不需要理解这些命令，日常只用双击启动或打开客户端窗口。

## 2. 工作人员每天只做的动作

1. 看「今日中控台」。
2. 对 `待生成` 点生成候选。
3. 对 `待选择` 选一条并锁定。
4. 对 `已锁定` 生成交付包。
5. 对 `可交付` 打开交付包，把文案和素材通过电脑微信发给兼职。
6. 发完后标记 `已派发`。
7. 兼职回传作品链接后点 `兼职已汇报`。
8. 进入 `待抖音核对`，打开抖音链接，回填播放、点赞、评论、粉丝和异常备注。
9. 看到爆款内容时，只需要在「爆款链接」粘贴链接。

## 3. 素材处理

新建案例后，系统会显示本地案例目录。把原始素材放入：

```text
00-原始素材/
```

然后回到案例详情：

1. 点「扫描素材」。
2. 把可用素材标记为 `可用`。
3. 如果任务进入 `素材阻塞`，先复制补素材说明，补齐后重新生成交付包。

大图片、大视频、交付包都放本地文件夹，数据库只保存路径和状态。

## 4. 微信交付规则

每个交付包优先看：

```text
00-微信发送清单.txt
```

发送顺序固定：

1. 发送 `01-发给兼职文案.txt`。
2. 发送 `02-抖音发布文案.txt`。
3. 按 `05-素材顺序清单.txt` 发送图片或视频。
4. 提醒兼职发布后回传作品链接或截图。

不要把交付包压缩后发给兼职；兼职端只接收普通文字、图片、视频。

## 5. 爆款链接和青豆

工作人员只负责粘贴链接。

Codex/AI 后续处理：

1. 从首页「待爆款分析」复制青豆任务。
2. 用青豆提取标题、正文/口播、评论区信息和结构。
3. 回填到爆款模板。
4. 按账号人设批量生成同结构内容。

链接未分析前不能批量生成，这是正常状态。

## 6. 本地 AI 顾问

首页有两个入口：

- `复制AI工作包`：生成 Markdown，并复制到剪贴板。
- `请本地AI建议`：生成工作包，并尝试调用本地 `claude`、`clude`、`claude-code` 或 `LOCAL_CLAUDE_COMMAND`。

命令行也可以直接执行：

```bash
cd /Users/licc/Desktop/素人系统/app
npm run consult
```

这个命令会调用当前运行中的服务；如果服务没开，会临时启动服务，生成工作包并尝试调用本地 AI 顾问。

工作包保存到：

```text
data/operator-packets/
```

顾问记录保存到：

```text
data/ai-consults/
```

如果本机没有装本地 AI 命令，系统会保存 `unavailable` 记录，不影响运营。

## 7. 抖音核对边界

当前默认做法是人工核对：

1. 系统打开抖音主页或作品链接。
2. 工作人员看发布时间、标题/正文、素材是否匹配。
3. 工作人员回填播放、点赞、评论、粉丝和异常备注。

后续可以做浏览器协同增强：客户端或浏览器插件尝试读取当前页面上的公开数据，再让工作人员确认。但它不能作为唯一方案，因为登录状态、页面改版、权限限制都会让自动读取失效。读不到时，仍按人工回填完成。

## 8. 交付前验收

每次改完系统后，在终端运行：

```bash
cd /Users/licc/Desktop/素人系统/app
npm run verify
```

如果要同时尝试本地 AI 顾问：

```bash
npm run verify:consult
```

如果要把 GitHub main 同步也作为硬性验收：

```bash
npm run verify:github
```

当前如果本机没有 GitHub 写权限，`verify:github` 会失败，这是预期提示。需要先完成 GitHub 认证或配置有写权限的 SSH key，再 push。

桌面客户端配置验收：

```bash
npm run client:check
```

## 9. 当前已知外部依赖

- `IMAGE_API_KEY` 为空：图片任务会显示“待填图片密钥”，不影响文案、排期、交付、核对。
- 本地 `clude/Claude` 未安装：AI 工作包和顾问记录仍可生成，但不会得到真实建议。
- GitHub 写权限未配置：本地系统可用，但远端 main 不会自动同步。
