# 素人系统日常运营 SOP

这份 SOP 面向每天实际使用系统的工作人员。

## 1. 每天启动

1. 双击 `启动素人工作台.command`。
2. 打开 `http://127.0.0.1:5174/`。
3. 进入「运行状态」，先看「系统验收清单」：
   - `已就绪`：可以正常使用。
   - `待接入`：通常是图片生成，不影响排期、文案、交付和抖音采集清单。
   - `需处理`：优先处理，例如访问码、目录或素材状态异常。

如果要用桌面窗口版，在终端运行：

```bash
cd /Users/licc/Desktop/素人系统/app
npm run client
```

生成 Mac 客户端目录：

```bash
npm run client:mac
```

Mac 客户端会用 Electron 内置 Node 启动本地后端，不需要工作人员手动启动服务。普通运营不需要理解这些命令，日常只用双击启动或打开客户端窗口。

如果工作人员在另一台电脑上使用，主机先在 `app/.env` 设置 `SOUREN_HOST=0.0.0.0` 和 `SOUREN_ACCESS_CODE`，重启后从「运行状态」复制局域网网址发给工作人员。工作人员只打开网址、输入访问码，不需要安装客户端。

## 2. 工作人员每天只做的动作

1. 看「今日中控台」里的「今日操作队列」，从上到下逐条处理；再看「后续账号预览」，确认今天涉及哪些兼职/账号。
2. 对 `待生成` 点生成候选。
3. 对 `待选择` 选一条并锁定。
4. 对 `已锁定` 生成交付内容。
5. 对 `可交付` 点「查看交付内容」，先确认当前只发给哪个微信，再按弹窗顺序复制文案、下载图片或视频，通过电脑微信发给兼职。
6. 发完后只在交付弹窗里标记 `已派发`。
7. 任务会进入折叠的「等待兼职确认」，不再挡住今日队列；确认兼职或剪辑人员完成发布/交付后，再展开等待确认并标记 `已完成`。
8. 如果首页出现爆款提醒，安排助理号/阑尾号去评论区互动，然后标记已安排互动。
9. 看到爆款内容时，只需要在「爆款链接」粘贴链接。

「后续账号预览」只是把操作队列按兼职归组，素材同步、剪辑、爆款互动和交付都会算进去；真正操作仍然只处理上方队首。

「异常账号」默认折叠，只是提醒今天有哪些账号有缺口或风险；不要展开后跳着处理，等它排到「今日操作队列」队首再动。

如果队列出现「补登记」，说明历史账号缺微信昵称、抖音主页、项目或共享原始素材路径。排到队首后打开案例，点「编辑案例」补齐资料；共享路径必须是已经存在的共享盘/服务器目录。

不要在「排期规划」或非队首案例卡片里抢先处理任务。它们只用来查看和定位；复制、派发、生成交付、剪辑完成、爆款互动这些动作必须等任务排到「今日操作队列」第一条后再做。已派发任务只在「等待兼职确认」里收尾，不再阻塞今日队列。不要批量准备一整天的交付内容，避免一下子制造多条待发送任务。

「排期规划」只用于看安排，不在里面生成、交付或打开案例；看到某条排期需要处理，也等它进入「今日操作队列」后再动。

「数据复盘」只用于看趋势和账号表现，不在里面处理具体案例；看到需要处理的账号，也等它进入「今日操作队列」后再动。

## 3. 素材处理

工作人员先把挑好的案例素材放到共享盘/服务器对应目录。登记案例有两种方式：

1. 如果这个目录已经在服务器待分配案例库里，进入「案例库」点「扫描服务器案例库」，找到未登记目录后点「登记案例」。
2. 如果是临时单个目录，新建案例时直接填写真实兼职微信昵称、抖音主页、项目和这个案例的「共享原始素材路径」。
3. 一次登记多个账号时用「批量导入」；每行都必须有微信昵称、抖音主页、项目和共享原始素材路径，系统会先整批检查，任何一行不通过都会整批不创建账号。

无论哪种方式，系统会先校验共享目录，并在登记时自动同步一次到本地受控副本。
登记成功或补登记编辑完成后，账号会自动进入采集队列；工作人员不用再去「运行状态」里手动登记采集。

删除误建案例时，必须在删除弹窗输入“删除 + 这个兼职的微信昵称”。确认后本地素材副本也会同步删除，不要把删除当成普通清理按钮。

```text
共享原始素材路径/挑好的图片或视频
```

然后回到案例详情：

1. 点「扫描素材」确认系统已经读到素材。
2. 把可用素材标记为 `可用`。
3. 如果共享目录后续又新增素材，再点「同步共享素材」，然后重新扫描。
4. 如果任务进入 `素材阻塞`，打开案例查看素材缺口，补齐后重新生成交付内容。

医院素材、套图素材、备用素材这类所有案例都可能用的内容，放到「运行状态」里显示的通用素材库，再点「扫描通用素材」。扫描后只在折叠的「通用素材校准」里检查分类、用途和可用状态。

扫描后先看素材用途是否正确。案例自己的素材一般标成 `案例人物`、`案例过程`、`案例对比` 或 `案例场景`；通用素材一般会自动分成 `医院素材`、`套图素材`、`生活场景`、`文字卡`。如果自动分类不准，案例素材在案例详情里改，通用素材在运行状态的「通用素材校准」里改。

做图片任务时，只从素材缺口或具体排期进入；不要临时新建没有用途的图片任务。系统会自动带入案例人物/变化参考和医院/通用场景素材；后面接 Image2 时，这些就是合成图的参考输入。

大图片、大视频和交付内容由系统保留后台副本，数据库只保存路径和状态；日常操作以网页内查看、复制、下载为主。

## 4. 微信交付规则

发送顺序固定：

0. 先看弹窗顶部“当前只发给谁”，微信窗口也只停在这个兼职上。
1. 第一次给这个兼职发内容时，先复制并发送“兼职须知”。
2. 在「查看交付内容」里复制“发给兼职文案”并发送。
3. 复制“抖音发布文案”并发送。
4. 按页面里的“素材发送顺序”下载/发送图片或视频。
5. 发完微信后，只在交付弹窗里点“已用微信发送”；这条会移到折叠的「等待兼职确认」。确认兼职已经按要求发布/交付后，再从等待确认里打开交付记录标记完成。

“兼职须知”和“发给兼职文案”已经写好发布时间窗、封面、话题、素材顺序和完成回复口径，不需要工作人员临场解释。

如果交付弹窗显示没有可发送图片或视频，先回去补素材或重新生成，不要把这条标记为已派发。

标记 `已派发` 后，交付弹窗只用于查看和完成确认，不再提供复制和下载按钮；需要重新发送时，先确认确实是同一个兼职的同一条内容。

进入交付链路后，不要再换候选稿或重新生成交付内容；系统会锁住这两类动作，避免已发内容被覆盖。

不要把本地交付文件压缩后发给兼职；兼职端只接收普通文字、图片、视频。

如果是视频内容，页面会显示口播/字幕文案、固定剪辑配方和「本次视频素材」。工作人员只发送这一组素材，按配方替换素材，剪辑或安排发布完成后，把剪辑任务标记为完成。

剪辑任务只从具体排期创建；同一条排期已经显示「剪辑任务已建」时，继续处理已有任务即可。

## 5. 爆款链接

工作人员只负责粘贴链接。

待分析链接处理流程：

1. 工作人员只粘贴链接，不填写模板内容。
2. 主机侧用 Chrome/轻抖等方式提取标题、正文/口播、评论区信息和结构。
3. 主机侧把分析结果写回 `/api/viral-templates/:id/analysis-result`。
4. 系统记录分析结果后，按账号匹配规则批量生成同结构内容。

链接未分析前不能批量生成，这是正常状态。

待分析爆款链接不会进入「今日操作队列」。分析完成后，只给可投放账号生成同款候选；暂停、休眠、不符合投放频率或已有同日任务的账号会自动跳过。

## 6. 抖音数据监控

工作人员不需要逐条手动录入播放、点赞、评论和粉丝。

1. 新建案例时填写抖音主页。
2. 系统按账号活跃度登记采集任务：正在投放或起量的账号勤采，偏冷账号降频，休眠账号一周一次。
3. 主机后台会自动用已登录抖音的 Chrome 按清单打开主页，并把账号快照和作品快照写回后台接口。`npm run collect:douyin -- --register --limit 10` 只作为排查命令保留；只想先看解析结果时运行 `npm run collect:douyin -- --dry-run --limit 3`。
4. 系统自动统计初始粉丝、最新粉丝、作品播放、点赞和评论。
5. 首页只展示工作人员需要处理的账号策略动作，例如爆款互动、偏冷补内容或增长承接；偏冷和增长动作可以直接生成下一条排期。
6. 如果作品达到爆款阈值，首页会提醒安排助理号/阑尾号互动。

当天还没采集时，状态只放在「运行状态」的折叠「系统后台」里；系统不会伪造数据，也不会把待采集当成工作人员首屏任务。

「运行状态」里的「系统后台」默认折叠，用来集中放待分析爆款、到期 Chrome 采集和图片任务；到期账号由后台自动排队并在主机自动执行采集，工作人员不需要把它当成今日任务，也不需要手动登记采集。

Chrome 采集执行器解析不到真实粉丝、作品或互动数据时会跳过，不会写入空结果。这样宁可保持“等待采集”，也不制造假的账号状态。

排期也按采集结果减量：连续低播放账号会自动改为两天一条，连续多条不起色时暂停自动补新排期；已生成候选、已锁定或可交付但还没派发的旧任务会同步撤下，已派发或已完成的历史记录不受影响。

如果某个账号已派发超过 3 天仍未确认完成，系统会按「失联暂停」自动处理：普通队列撤下，采集暂停，不再为这个账号生成新的首页待办。

如果兼职后来又回来配合，在案例详情点击「恢复账号」。系统会把旧的超时派发任务标为「已取消」，并补新的近期排期。

## 7. 交付前验收

每次改完系统后，在终端运行：

```bash
cd /Users/licc/Desktop/素人系统/app
npm run verify
```

如果要把远端主分支同步也作为硬性验收：

```bash
npm run verify:github
```

`verify:github` 会检查本地提交是否已经同步到远端主分支；如果失败，先看终端提示，再补做认证、提交或推送。

桌面客户端配置验收：

```bash
npm run client:check
```

## 8. 当前已知外部依赖

- 图片生成未接入：图片任务会显示“待接入图片接口”，不影响文案、排期、交付和账号监控。
- 远端同步异常：本地系统可用，但需要先按 `verify:github` 的提示处理认证、提交或推送。
